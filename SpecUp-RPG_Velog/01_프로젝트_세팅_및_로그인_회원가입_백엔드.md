# [SpecUp RPG] 01. 프로젝트 세팅 & 회원가입/로그인 백엔드 구현

## 구현 목표

- Spring Boot + Neon(PostgreSQL) 환경 세팅
- JWT 기반 회원가입 / 로그인 API 구현
- 공통 응답 형식(`ApiResponse`) 및 예외 처리 구조 설계

---

## 기술 선택 이유

### Neon (PostgreSQL) 선택

처음엔 로컬 MariaDB로 개발을 시작하려 했다. 하지만 개발 환경에서 MariaDB를 쓰고 배포 환경에서 PostgreSQL(Neon)으로 바꾸면 아래 문제가 생긴다.

- `AUTO_INCREMENT` → `SERIAL` 변환
- 날짜 함수 차이
- Dialect 변경

나중에 배포할 때 이런 것들을 하나씩 수정해야 하는 번거로움이 생기기 때문에, **처음부터 Neon(PostgreSQL)** 을 쓰기로 결정했다.

### JWT 선택

세션 방식은 서버에 상태를 저장해야 해서 확장이 어렵다. JWT는 토큰 자체에 유저 정보를 담아 **Stateless** 하게 인증할 수 있어서 REST API에 적합하다.

## 핵심 코드 설명

### 1. 공통 응답 형식 — ApiResponse

모든 API 응답을 아래 형식으로 통일했다.

```json
{
  "success": true,
  "data": { },
  "message": "요청이 성공했습니다."
}
```

```java
@Getter
public class ApiResponse<T> {
    private final boolean success;
    private final T data;
    private final String message;

    public static <T> ApiResponse<T> success(T data, String message) {
        return new ApiResponse<>(true, data, message);
    }

    public static <T> ApiResponse<T> fail(String message) {
        return new ApiResponse<>(false, null, message);
    }
}
```

프론트에서 `response.data.success`로 성공 여부를 항상 동일한 방식으로 처리할 수 있다.

### 2. User 엔티티 — AI 토큰 시스템 포함

```java
public static User create(String email, String password, String nickname) {
    User user = new User();
    user.email = email;
    user.password = password;
    user.nickname = nickname;
    user.role = "DEVELOPER";
    user.aiToken = 10;
    user.aiTokenMax = 10;
    user.aiTokenResetAt = LocalDateTime.now().plusHours(24);
    return user;
}
```

Claude.ai처럼 시간이 지나면 자동으로 AI 토큰이 충전되는 구조를 설계했다. `aiTokenResetAt`이 지나면 Spring Scheduler가 토큰을 자동으로 충전한다.

### 3. 레벨업 로직 — UserStatus

```java
// 레벨업 공식: 다음 레벨 필요 XP = 현재 레벨 * 200
private boolean checkLevelUp() {
    int requiredXp = this.level * 200;
    if (this.experiencePoints >= requiredXp) {
        this.experiencePoints -= requiredXp;
        this.level++;
        return true;
    }
    return false;
}
```

Lv.1 → Lv.2는 200 XP, Lv.5 → Lv.6은 1000 XP가 필요하다. 레벨이 오를수록 더 많은 XP가 필요한 구조다.

### 4. JWT 인증 흐름

```
로그인 요청
    │
    ▼
AuthService.login()
    │ 이메일로 유저 조회
    │ 비밀번호 BCrypt 검증
    ▼
JwtProvider.generateToken()
    │ userId + email → JWT 토큰 생성
    ▼
토큰 반환 → 프론트에서 저장
    │
    ▼ (이후 모든 요청)
JwtFilter.doFilterInternal()
    │ Authorization: Bearer {토큰} 헤더 확인
    │ 토큰 유효성 검증
    │ SecurityContext에 인증 정보 저장
    ▼
컨트롤러 실행
```

### 5. 회원가입 시 자동 생성 처리

```java
@Transactional
public AuthResponseDto.Register register(AuthRequestDto.Register request) {
    // 이메일 중복 체크
    if (userRepository.existsByEmail(request.getEmail())) {
        throw new CustomException(ErrorCode.DUPLICATE_EMAIL);
    }

    // 유저 생성
    User user = User.create(request.getEmail(),
            passwordEncoder.encode(request.getPassword()),
            request.getNickname());
    userRepository.save(user);

    // 유저 스탯 자동 생성 (Lv.1, XP 0, 골드 0)
    UserStatus userStatus = UserStatus.createDefault(user);
    userStatusRepository.save(userStatus);

    return AuthResponseDto.Register.from(user);
}
```

회원가입 시 `users` 테이블과 `user_status` 테이블에 동시에 데이터가 생성된다. `@Transactional`로 묶어서 하나라도 실패하면 전체가 롤백된다.

## 트러블 슈팅

### 1. 시스템 환경변수가 application.properties를 덮어쓰는 문제

**증상**: `application.properties`에 Neon 주소를 설정했는데도 계속 다른 DB 주소로 접속 시도

**원인**: Spring Boot는 설정 우선순위가 있다.
```
1순위: 시스템 환경변수 (SPRING_DATASOURCE_URL 등)
2순위: application.properties
```
이전 프로젝트에서 등록해둔 시스템 환경변수가 남아있어서 계속 덮어쓰고 있었다.

**해결**: IntelliJ Edit Configurations → Environment variables에 Neon 정보를 직접 입력해서 시스템 환경변수를 덮어썼다.

**교훈**: 새 프로젝트를 시작할 때는 시스템 환경변수에 `SPRING_DATASOURCE_*` 관련 변수가 없는지 먼저 확인하자.

---

### 2. column "password" does not exist 에러

**증상**: 회원가입 API 호출 시 `ERROR: column "password" of relation "users" does not exist`

**원인**: `schema.sql`에서 컬럼명을 `password_hash`로 생성했는데, `User.java` 엔티티에서 필드명을 `password`로 선언해서 JPA가 `password` 컬럼을 찾지 못함

**해결**: Neon SQL Editor에서 컬럼명을 수정
```sql
ALTER TABLE users RENAME COLUMN password_hash TO password;
```

**교훈**: 엔티티 필드명과 실제 DB 컬럼명을 일치시키거나, `@Column(name = "password_hash")`로 명시적으로 매핑해야 한다.

## 느낀 점 / 배운 점

- Spring Boot의 설정 우선순위를 몰라서 엄청 헤맸다. 환경변수가 `application.properties`보다 우선순위가 높다는 걸 이번에 확실히 배웠다.
- 엔티티 설계할 때 DB 컬럼명과 필드명을 처음부터 맞춰두는 게 얼마나 중요한지 깨달았다.
- `@Transactional`로 회원가입 시 users + user_status를 원자적으로 처리하는 구조가 중요하다는 걸 배웠다.
